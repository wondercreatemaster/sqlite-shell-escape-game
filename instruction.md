Our little "shell-escape" puzzle game is backed by a SQLite database at `/app/game/escape.db`, and the seeded config is a security and integrity mess: hints leak answers in plaintext, command strings are full of stray whitespace, inline comments, shell quoting, and backslash escapes, rooms are all unlocked regardless of whether they're actually reachable, wildcard/`admin` "unlock everything" commands are present, challenges and doors point at unsafe or unrunnable commands, `challenge_checks.points` are stale zeros, the per-room `cmd_digest` values are stale, and the signed `meta` manifest is a placeholder. Harden the database in place. The database has exactly these five tables and columns (keep this schema and column order unchanged — add/drop no tables or columns):

```
rooms(id INTEGER PK, name TEXT, description TEXT, hint TEXT, is_locked INTEGER, cmd_digest TEXT)
allowed_commands(id INTEGER PK, room_id INTEGER, command TEXT)
challenge_checks(id INTEGER PK, room_id INTEGER, expected_command TEXT, points INTEGER)
doors(id INTEGER PK, from_room INTEGER, to_room INTEGER, via_command TEXT)
meta(key TEXT PK, value TEXT)
```

You must parse the command strings with a correct POSIX-style quote/escape lexer. Scan left to right tracking single-quote (`'`) and double-quote (`"`) state. Outside any quotes, a backslash `\` escapes the very next character, making it an ordinary literal (so `\|`, `\#`, `\*`, `\?`, and `\ ` are literal and never act as a pipe, comment, glob, or separator, and `\'`/`\"` do not start a quote); a trailing backslash at end of string is a literal backslash. Inside single or double quotes, backslash is an ordinary literal character and a quote of the other kind is literal; a `'` inside single quotes and a `"` inside double quotes close that quote. **Canonical form** of a command: an unquoted, unescaped `#` that is at the start of the string or immediately preceded by unquoted, unescaped whitespace begins a comment — drop it and everything after; then collapse every run of unquoted, unescaped whitespace to a single space and strip such leading/trailing whitespace. Quote characters, escape backslashes, and the bytes they protect are all kept verbatim. **Programs of a command**: split the canonical command into pipeline stages on each unquoted, unescaped `|`; in each stage split into tokens on unquoted, unescaped whitespace, then remove quotes and resolve escapes (`\X` → `X`) to get each token's literal value; drop leading `VAR=value` env-assignment tokens; if the first remaining token's basename is `env`, drop that `env` plus any following `-option` or `VAR=value` tokens; the stage's program is the basename (path stripped) of the next token. A command "contains a glob" iff it has an unquoted, unescaped `*` or `?`.

The only trusted programs are the GNU coreutils utilities; the coreutils 9.5 source tarball is staged at `/app/coreutils-9.5.tar.gz` (no network) and the utility names are listed in its top-level `README`.

A binary policy blob is staged at `/app/game/policy.bin` and you must parse it (all multi-byte integers are big-endian): bytes 0–3 are the ASCII magic `SESC`; byte 4 is the format version (`2`); bytes 5–6 are a `uint16` digest round count `R`; bytes 7–8 are a `uint16` key length `K`; the next `K` bytes are the ASCII HMAC key used in rule 9; then a `uint16` count `N`; then `N` entries, each a single `uint8` length followed by that many ASCII bytes naming an extra denylisted program; then a trailing `uint32` CRC-32 (IEEE 802.3, as in zlib / `java.util.zip.CRC32`) computed over every byte before it (the staged blob is valid; validating this checksum is optional). The **effective denylist** for rule 3 is `rm, dd, shred, chroot, runcon` together with the extra program names from the policy; `R` is the digest round count for rule 8; and the HMAC key for rule 9 is the key bytes from the policy.

Apply these rules to `/app/game/escape.db`, ordered to respect their dependencies:

1. Rewrite `allowed_commands.command`, `challenge_checks.expected_command`, and `doors.via_command` to their canonical form. Every comparison, count, length, and hash below uses these canonical strings.
2. Set every `rooms.hint` to SQL `NULL`.
3. In `allowed_commands`, keep a row only if its command contains no glob and **every** program of its command is a coreutils utility from the README that is not in the effective denylist (base denylist plus the policy's extra names). Delete the rest.
4. In `challenge_checks`, delete any row whose `(room_id, expected_command)` does not exactly equal a surviving `allowed_commands` `(room_id, command)`.
5. For every surviving `challenge_checks` row, set `points` to `7 * S + L`, where `S` is the number of pipeline stages of its canonical `expected_command` and `L` is that string's length in characters.
6. In `doors`, a door is *usable* iff its `via_command` exactly equals a surviving `allowed_commands.command` in the door's `from_room`. Delete every non-usable door.
7. Set `rooms.is_locked` to `0` if the room is reachable from room `1` by following surviving doors in the `from_room` → `to_room` direction (room `1` is reachable by definition), otherwise `1`.
8. Build each room's message as its surviving `allowed_commands.command` values, sorted ascending by byte value and joined with a single `\n` (no trailing newline; a room with no surviving commands uses the empty string). Set `rooms.cmd_digest` to the **`R`-times-iterated** SHA-256 of that message: start from the UTF-8 bytes of the message and, `R` times, replace the value with the raw 32-byte SHA-256 digest of the current value; `cmd_digest` is the lowercase hex of the final 32 bytes (`R` comes from the policy).
9. Build the manifest message by joining, for every room ordered by `id`, the line `<id>=<cmd_digest>` with `\n` (no trailing newline). Compute HMAC-SHA256 of that message with the key parsed from the policy, and set the `meta` row with key `manifest` to the **RFC 4648 Base32** encoding (standard `A–Z2–7` alphabet, uppercase, with `=` padding) of the raw 32 HMAC bytes.

Don't alter the table schemas, don't renumber any rows, and leave every other column and row exactly as seeded.
